### 各ファイルの詳しい役割

#### 1. `main.py`
これが**システムの本体**です。これまでのチャットで作ってきたコードはすべてここに書かれています。

*   **何をしているか？**
    *   RSSからニュースを集める。
    *   「上方修正」などのキーワードで選別する。
    *   GPT-5.2に分析させる。
    *   Slackに通知し、Google Sheetsに保存する。
*   **変更する場所:**
    *   キーワードを追加・変更したい時

#### 2. `requirements.txt`
システムを動かすために必要な「部品（ライブラリ）」のリストです。

*   **中身:**
    `openai`, `slack_sdk`, `feedparser` などが箇条書きされています。
*   **役割:**
    GitHubのサーバーが起動した際、最初にこのリストを見て「この部品が必要なんだな」と理解し、自動でインストールを行います。
*   **変更する場所:**
    基本的には触りません。

#### 3. `sources.json`
データの元となるRSSのリストです。

*   **中身:**
    決算関連の資料があるRSSのURLが箇条書きされています。
*   **役割:**
    GitHubのサーバーが起動した際、このリストからデータを収集します。
*   **変更する場所:**
    サイトや企業を変更したいときに編集します。

#### 4. `.github/workflows/run_schedule.yml`
これが**24時間自動稼働のための**「**指令書（タイマー設定）**」です。

*   **場所の注意点:**
    必ず `.github` フォルダの中の `workflows` フォルダの中になければ動きません。（ドット `.` から始まることに注意）
*   **中身:**
    *   `cron: '*/30 * * * *'` ➡ 「30分おきに起動せよ」という設定。
    *   「Secrets（APIキー）を読み込んで `main.py` を実行せよ」という手順。
*   **変更する場所:**
    *   「1時間に1回」や「朝9時だけ」など、**頻度を変えたい時**にここを編集します。

### 【マニュアル】OPENAI_API_KEY とは？

今回構築したシステムでは、文章を分析するためにOpenAI社の「GPT」を使用します。
`OPENAI_API_KEY` は、AIを利用するために必要な「**デジタル通行手形**」のようなものです。

#### 1. 何に使うもの？
GitHubにあるプログラムが、OpenAI社のAIに接続する際の証明をするために使います。これがないと、システムはAIを動かすことができず、分析ができません。

#### 2. 料金について（重要）
このキーを使うと、AIが文章を読んだり書いたりする量に応じて「**従量課金（使った分だけ支払い）**」が発生します。

*   **月額20ドルの「ChatGPT Plus」とは別物です。**
    *   ChatGPT Plus（チャット画面）は、人間が使うための定額制プランです。
    *   API（今回のシステム）は、プログラムが使うための従量制プランです。別途クレジットカードの登録が必要です。
*   **費用感の目安:**
    *   今回のシステムで、決算書1通をGPT-5.2で分析する場合、おおよそ数銭〜数円程度のコストがかかります（文章量によります）。
    *   使わなければ0円です。

#### 3. 取得方法
まだお持ちでない場合は、以下のOpenAI公式ページから発行できます。
1.  [OpenAI Platform](https://platform.openai.com/) にアクセス・ログインする。
2.  「Settings」→「Billing」でクレジットカードを登録する（※ここが必須です）。
3.  「API Keys」のメニューから「Create new secret key」を押して発行する。
    *   `sk-...` から始まる長い文字列が発行されます。これがキーです。

### 【マニュアル】SLACK_WEBHOOK_URL とは？

`SLACK_WEBHOOK_URL` は、今回作成したAIシステムから、Slackへメッセージを届けるための「**専用の郵便ポストの住所**」のようなものです。

#### 1. 何に使うもの？
このシステムはクラウド（GitHub）上で動いていますが、分析が終わった瞬間に、結果をスマートフォンやPCのSlackに通知する必要があります。
「このURLにデータを送ると、Slackの『◯◯チャンネル』にメッセージが表示される」という仕組みになっています。

#### 2. どんな見た目？
`https://hooks.slack.com/services/...` から始まる、非常に長い英数字の文字列です。
これが「システムとSlackをつなぐ直通電話番号」の役割を果たします。

#### 3. 取得方法（発行の手順）
このURLは、ご自身のSlack管理画面から無料で発行できます。以下の手順で行ってください。

1.  PCのブラウザで [Slack API: Your Apps](https://api.slack.com/apps) にアクセスします。
2.  **「Create New App」** ボタンを押し、**「From scratch」** を選びます。
    *   **App Name:** 何でもOKです（例: `Stock-Bot`）
    *   **workspace:** 通知を送りたいSlackワークスペースを選択
    *   「Create App」ボタンを押します。
3.  左側のメニューから **「Incoming Webhooks」** をクリックします。
4.  右上のスイッチを **「On」** に切り替えます。
5.  画面下部に出現する **「Add New Webhook to Workspace」** ボタンを押します。
6.  通知を送りたい「チャンネル（例: #general や #決算速報）」を選び、「許可する」を押します。
7.  画面に **「Webhook URL」** が表示されますので、「Copy」ボタンを押してコピーしてください。

このコピーしたURLを、GitHubの設定画面（Secrets）に登録します。

## システム概要

このリボリトジは、「**企業の決算情報や開示ニュースを自動収集し、AIで分析してSlackに通知・Googleスプレッドシートに記録する自動監視システム**」です。

機関投資家のような視点で、株価に影響がありそうなニュースをフィルタリング・要約することを目的としています。

処理の具体的な流れは以下の通りです。

### 1. 準備・初期化
*   **環境変数の確認**: OpenAI、Slack、Google認証情報のAPIキーが設定されているか確認します。
*   **履歴の読み込み**: Googleスプレッドシートに接続し、過去に処理済みのURL一覧を取得します（同じニュースを二重に通知しないため）。

### 2. 情報収集 (RSS取得)
*   **対象サイト**: PR TIMESやJPXのRSSフィードから最新情報(3日以内と過去のデータ(5年以内)を取得します。
*   **一次フィルタ**: 記事タイトルに「決算」「修正」「増配」などのキーワード（`TARGET_KEYWORDS`）が含まれるものだけを抽出します。

### 3. コンテンツの抽出 (スクレイピング)
*   各ニュースのURLにアクセスし、本文を取得します。
*   **PDF対応**: `pdfplumber` を使用して、決算短信などのPDFファイルからもテキストを抽出します。
*   **HTML対応**: `BeautifulSoup` を使用して、Webページの記事本文を抽出します。
*   **ブロック回避**: サーバーにブロックされないよう、ブラウザのふりをするヘッダー（User-Agent）を付与し、リトライ処理も行います。

### 4. キーワード判定 (二次フィルタ)
*   抽出したテキスト内に、ポジティブワード（「上方修正」「増配」など）やネガティブワード（「下方修正」「赤字」など）が含まれているかチェックします。
*   キーワードが一つもない場合は、関係ない記事としてスキップします。

### 5. AIによる分析 (OpenAI API)
*   キーワードにヒットした記事を、ChatGPT (GPTモデル) に送信します。
*   **プロンプト（指示）**: AIに対し「プロの機関投資家」として振る舞うよう指示し、以下のJSON形式で分析結果を出力させます。
    *   **判定**: 強気 / 中立 / 弱気 / 要警戒
    *   **根拠**: なぜそう判断したか
    *   **要約**: 3行以内のサマリー
    *   **インパクト**: 株価への短期的な影響度

### 6. 結果の出力
*   **Slack通知**: 分析結果を見やすく整形してSlackに送信します。
    *   「ポジティブ/強気」なら緑色、「ネガティブ/弱気」なら赤色で装飾されます。
    *   原文へのリンクボタンも付きます。
*   **ログ保存**: 分析日時、タイトル、判定結果、要約などをGoogleスプレッドシートに追記します。

### 主な特徴
*   **PDFが読める**: 決算短信PDFの中身まで読んで判断できる点が強力です。
*   **AIによる定性判断**: 単なるキーワード検索だけでなく、文脈をAIが読んで「強気」「弱気」を判断します。
*   **重複排除**: 一度処理したURLはスプレッドシートを参照してスキップします。

### 補足（GitHub Actionsの仕様について）

GitHub Actionsというプラットフォームの特性上、以下の2点だけ頭の片隅に置いておいてください。

1.  **実行タイミングのズレ**
    `cron: '*/30 ...'` と設定しても、GitHub側のサーバー混雑状況によっては **数分〜15分程度開始が遅れる** ことがよくあります。（無料枠では仕様です）。
    *   **対策:** コード側で `TIME_THRESHOLD = timedelta(hours=3)` と余裕を持たせているので、**遅れて実行されても取りこぼしは発生しません。** そのための設定ですのでご安心ください。

2.  **「60日ルール」による自動停止**
    GitHub Actionsは、リポジトリに対して60日間誰もコミット（更新）をしないと、**「使われていない」と判断されてスケジュール実行が自動的に停止**します。
    *   **対策:** 停止する前にGitHubからメールが届きます。その際は、リポジトリ画面にある「Enable workflow」ボタンを押すか、何でもいいので1行変更してコミットすれば再開されます。

この構成で、30分ごとの自動監視システムとして問題なく稼働します。
